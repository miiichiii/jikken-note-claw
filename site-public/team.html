Mission Control Routing

センセイ enters by DM, macstudio-bot routes the work.

This page is the operating map for Mission Control: one primary front door, one orchestrator in the middle, and clearly separated execution lanes for deep sub-agent work, light drafting/checking, simple local tasks, and Thunderbird draft execution.

Live posture

Primary intake DM
Main model GPT-5.4
Heavy lanes Sub-agents
UI execution Thunderbird

Keep the chat clean, route heavy work outward, and reserve cheap/simple paths for bounded jobs only.

Routing policy strip

Prefer GPT-5.4-class or Gemini 3.1 Pro-class sub-agents for coding, research, reviews, and long-running work. Use gemini-3-flash-preview only for light drafting or quick checks. Use local AI only for truly simple bounded tasks. Treat Thunderbird draft saving as a separate UI-execution lane, not a general reasoning lane.

Operating Principles

One front door

DM stays the primary intake so approvals, clarifications, and results do not scatter across channels.

Delegate depth

When work becomes long, technical, or research-heavy, push it to stronger sub-agents instead of overloading the orchestrator.

Use cheap paths carefully

Flash and local models are useful, but only when the task is clearly small, bounded, and low-risk.

Separate reasoning from execution

Thunderbird draft saving belongs to an execution lane so UI actions remain explicit and controllable.

Quick routing guide

Build feature / inspect repo / deep review
Route to a coding or research sub-agent with GPT-5.4-class or Gemini 3.1 Pro-class capacity.
Quick draft polish / small wording check
Use gemini-3-flash-preview if the task is lightweight and disposable.
Tiny bounded local helper task
Use local AI only if the request is genuinely simple and does not need stronger reasoning.
Save a prepared email draft in Thunderbird
Use the Thunderbird execution lane rather than treating it like a normal model step.
